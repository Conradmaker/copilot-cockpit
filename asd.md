- Mate가 생각해보니 직접 수정이 아니라 hanoff 하게되면 다른 에이전트를 호출하게 되는 구조였어. 근데 그 에이전트가 생각해보니 vscode copilot 기본 에이전트네..  그 에이전트가 구현 단계를 이끌어 나가면서 계속 수정을 한거였던거야. (새로운 워크로드를 고민하고 있어.)
- 플랜 모드에서는 Mate와 Coordinator가 의견을 주고 받았으면 좋겠어. 그리고 나에게 의견을 취합한뒤 askQuestions를 하고, 점점 브레인 스토밍 하듯이 계획안의 스펙을 뚜렷하게 정의해 나가는거야. 이 과정에서 에이전트의 판단하게 병렬로 서브에이전트들을 호출해서 자료조사를 해도 상관 없어.  대신 완성된 플랜은 handoff시에 다른 에이전트에게 이관되기 때문에 step이 세분화되고 요구사항과 스펙 그 외의 모든게 명확해야해. 
- 그러려면 coordinator는 조금 더 기획자, 디자이너의 관점에서 프로덕트나 기능을 분석하고 플랜을 검토하는 부분에 치중되어야 겠다. 
- Mate가 Coordinator 서브에이전트를 2개 병렬로 돌려서 각각의 의견을 들어도 괜찮을 것 같아. 모델이 다르면 의견이 달라질 수 있잖아. 
- Coordinator가 정리한 askQuestions 후보를 바로 Mate가 나에게 전달하기 보다는 만약 Coordinator가 병렬이었다면 여러 의견을 취합해서 Mate의 판단으로 나에게 질문하는게 좋을 것 같아. 
- 이러는 과정중에 Mate와 Coordinator는 플랜은 계속해서 다듬어 나가는거야. Mate는 Coordinator의 의견을 반영해서 계획안을 수정할 수도 있고, Coordinator는 Mate가 수정한 계획안을 검토하면서 또 의견을 줄 수도 있고. 이런 식으로 계속 브레인 스토밍 하듯이 계획안의 스펙을 뚜렷하게 정의해 나가는거야. 그러는 중에 이 플랜이 작업페이즈로 핸드오프되었을 경우 작업을 어떻게 진행하게 될지 판단하게 되는데, 하나는 Amp같이 하나의 작업자가 메인 에이전트가 되어서 작업 페이즈를 끝까지 완성하는 방식이 있고, 다른 하나는 Coordinator가 작업 페이즈에서 여러 작업자들을 관리하는 방식이 있을 것 같아.
- 이 두가지의 작업방식중 Mate와 Coordinator는 플랜을 정하면서 사용자인 나에게 추천하는 방식을 추천해주는게 좋을 것 같아. 
- 두 작업방식을 선택하는 기준이 명확해야 하는데, 작업의 규모로 판단하는게 아닌, 작업의 규모 + 플랜과 스펙상에 있는 계획을 간섭이 안될정도로 나눠서 확실하게 작업이 가능한지 여부로 판단하는게 좋을 것 같아. 만약 단일 기능이 복잡해서 컨택스트와 배경이 중요해질 경우에는 Amp같이 하나의 똑똑한 에이전트로 밀어붙이는게 더 이득이니까.
- 이에 따라 Mate.agent.md의 implementation handoff 프롬프트도 수정하고 handoff에는 Fleet Mode, Rush Mode, Open in Editor 3가지 옵션을 주는게 좋을 것 같아. 이에 따른 프롬프트도 적용해줘.
- 내가 implementation handoff를 선택하면 이제 Mate의 역할은 끝나. 그렇게 때문에 Mate는 Coordinator와 자료들, 그리고 나의 의견들을 조화롭게 해서 완벽한 계획과 스펙을 만들어내는것에 중점이 있는거지. 
- Mate에게서 떠난 계획안은 이제 oh-my-opencode의 atlas같은 boulder execution orchestrator. 즉, 계획 기반 실행과 verification gate에 특화되어 있고 구현자라기보다 conductor에 가까운 에이전트로 (Coordinator) 이관될거야. 그러니까 Coordinator는 Mate가 만들어낸 계획안을 바탕으로 실행 계획을 세우고, 필요한 경우 추가 자료조사도 하고, 구현이 제대로 되고 있는지 검토하는 역할이 될거야. 팀에서 가장 똑똑한 기획자 + PM이 되는거라고 생각하면 될 것 같아.
- 백그라운드 실행까지는 필요 없어.
