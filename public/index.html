01· 문제 정의

채팅으로 끝난 작업은
다시 읽히지 않습니다.

Copilot Chat은 강력하지만 대화는 쉽게 흩어집니다. 같은 결정을 다시 설명하게 되고, 역할 경계 없이 한 응답에 책임이 몰립니다. 그렇다고 매 단계마다 훅으로 강제하면 컨텍스트가 길어지고, 컨텍스트가 길어질수록 모델은 오히려 둔해집니다.

  • 01

    AI가 추측으로 답해도 그걸 바로 알아차리기 어렵습니다.

    코드베이스를 보지 않고 그럴듯한 코드만 내놓으면, 검증 부담은 결국 사람이 떠안게 됩니다.

  • 02

    요구사항, 설계, 구현, 리뷰가 한 채팅창에 뒤섞입니다.

    단계 경계가 없으면 PRD가 잠기기 전에 구현이 시작되고, 리뷰 관점도 한 에이전트에 몰립니다.

  • 03

    결과물이 채팅 로그에만 남아 다음 작업으로 이어지지 않습니다.

    다시 읽고 판단할 산출물이 없어서 같은 맥락을 매번 처음부터 설명하게 됩니다.

02· 워크플로

5단계가 아니라
결과물이 쌓이는 5개의 게이트.

각 단계는 채팅 로그가 아니라 사람이 다시 읽을 수 있는 .md 산출물을 남깁니다. 승인된 결과물이 있어야 다음 게이트가 열립니다.

  1. 01· 계획

    planning

    Mate가 요구사항을 정리하고 PRD를 잠급니다. Coordinator가 여러 관점에서 점검한 뒤에야 다음 단계로 넘어갑니다.

    ↳prd.md
  2. 02· DESIGN

    ui & ux

    Designer가 화면, 인터랙션, 톤을 정리합니다. PRD를 다시 쓰지 않고 구현 직전까지 디자인 결정을 잠급니다.

    ↳design.md
  3. 03· 기술

    architecture

    Architector가 스택, 경계, 데이터 계약을 정의합니다. 추측이 아니라 근거 있는 기술 결정으로 구현을 시작합니다.

    ↳technical.md
  4. 04· 구현

    implementation

    Commander와 Deep Execution이 분리된 범위 안에서 구현하고, 변경 근거와 검증 기록을 남깁니다.

    ↳execution-plan.md
  5. 05· 리뷰

    verdict

    Reviewer가 보안, 성능, 디자인, 계약 관점에서 최종 판정을 내립니다. 통과해야만 마무리 단계로 넘어갑니다.

    ↳review.md

03· 핵심 차이

채팅이 아니라 시스템입니다.

같은 모델이라도 운영 방식이 결과를 바꿉니다. 채팅창과 하네스의 차이를 한 화면에서 바로 비교해 보세요.

채팅창 −

결과물 한 번 쓰고 사라지는 응답
흐름 단계 경계 없이 한 창에 뒤섞임
재현성 맥락을 매번 다시 설명해야 함
책임 한 에이전트가 모든 결정을 떠안음
VS

하네스 +

결과물 다시 읽을 수 있는 .md 산출물
흐름 phase 단위 게이트로 잠긴 단방향 진행
재현성 산출물이 다음 작업의 입력이 됨
책임 역할별 에이전트가 분리된 관점으로 검증
F01 · RETRIEVAL-LED

추측보다 조회

구현 전에 코드베이스, 공식 문서, 로컬 패턴을 먼저 조회하도록 에이전트 행동을 유도합니다. 그럴듯한 코드보다 근거 있는 코드를 남깁니다.

↳.github/instructions/
F02 · EXPLICIT GATES

승인 없이 다음 단계로 가지 않는다

승인된 PRD가 없으면 구현이 열리지 않고, 리뷰를 통과해야만 마무리 단계로 넘어갑니다. 단계 경계가 분명합니다.

↳.github/agents/workflows/
F03 · ROLE-BASED REVIEW

여러 관점의 리뷰를 동시에

보안, 성능, 디자인, 계약을 각 리뷰 역할이 병렬로 살핍니다. 한 에이전트에 모든 판단을 몰아주지 않습니다.

↳.github/agents/reviewer-roles/

05· FAQ

자주 묻는 것들,
짧게 모았어요.

이 보일러플레이트는 워크플로를 강제하지 않습니다. 똑똑한 메인 에이전트의 판단을 존중하고, 시스템은 인덱스와 게이트만 제공합니다.

Q01 왜 워크플로를 강제하지 않나요? +

강제 훅은 컨텍스트 낭비입니다. 매 작업마다 모든 스킬을 읽게 만들면 컨텍스트가 길어지고, 컨텍스트가 길어질수록 모델은 더 둔해집니다.

대신 인덱스로 “여기 이런 스킬이 있다”고만 알려주고, 메인 에이전트가 상황에 맞게 직접 고르도록 둡니다. 모든 룰에는 왜가 먼저 적혀 있어 모델이 스스로 판단할 수 있습니다.

Q02 단계마다 커밋이 쌓이지 않나요? +

쌓지 않습니다. 요즘 도구는 되돌리기가 충분히 강력합니다. 단계마다 커밋을 연발하는 대신, 사람이 다시 읽기 좋은 산출물(.md)만 게이트마다 남깁니다.

결국 중요한 건 사람이 알아보기 편한 코드와 결정 기록입니다.

Q03 서브에이전트를 깊게 중첩하지 않는 이유는요? +

2뎁스 이상은 지양합니다. AI가 만든 프롬프트가 다시 다른 AI로 넘어가는 동안 누락되는 컨텍스트가 결과물의 신뢰도를 떨어뜨리기 때문입니다.

인수인계는 짧게, 필요한 정보만. 현실의 인수인계도 탕비실 정보까지 넘기지는 않습니다.

Q04 중대한 결정은 어떻게 검증하나요? +

필요한 순간에만 council을 소집합니다. 일상적인 작업까지 council을 끌어오지 않습니다.

council은 배경 컨텍스트를 최소화하고, 가능하면 제3의 CLI를 끌어와 중립성을 유지합니다. 같은 모델끼리만 합의하면 같은 편향을 강화하기 때문입니다.

Q05 MCP가 거의 없는 이유는요? +

MCP는 최소화합니다. 도구가 늘수록 컨텍스트가 길어지고, 컨텍스트가 길어질수록 모델은 더 바보가 됩니다.

대신 retrieval-led 원칙으로, 필요한 순간에 코드베이스와 공식 문서를 직접 조회하도록 유도합니다.

Q06 Plan 모드와 어떻게 어울리나요? +

요즘 CLI/에이전트 코딩 도구는 모두 plan 모드를 제공합니다. 이 하네스는 plan 모드에서 최대 효율이 나도록 게이트와 산출물 구조를 맞췄습니다.

목표는 기존 도구의 기능을 가리는 것이 아니라, 그 도구의 능력을 120%까지 끌어내는 것입니다.

Q07 모든 게 AGENTS.md에서 시작한다고요? +

시작이자 끝입니다. 워크플로, 스킬 인덱스, 서브에이전트 계약은 모두 AGENTS.md에서 출발해 다시 그곳으로 수렴합니다.

당연한 정보는 넣지 않습니다. AI는 React 코드를 보고 Vue라고 할 정도로 어리석지 않으므로, 룰은 모델이 실제로 판단할 때 필요한 핵심만 담습니다.
