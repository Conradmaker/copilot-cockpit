Why Agent Harness

AI 코딩, 혼자 쓰면 왜 한계가 있을까요?

단일 AI에 기획, 구현, 리뷰를 전부 맡기면 생기는 구조적 문제들이 있습니다. 하네스는 이 문제를 역할 분리와 artifact handoff로 해결합니다.

⛔️

대화가 길어질수록 품질이 떨어진다

하나의 대화가 이어지면 AI는 이전 문맥에 오염됩니다. 결정이 덮이고, 앞서 정한 방향이 조용히 무시되기 시작합니다.

→ Role-scoped 에이전트가 독립 컨텍스트로 각 단계를 처리합니다
🔀

기획, 설계, 구현, 리뷰를 모두 혼자 맡는다

단일 AI가 모든 역할을 처리하면 각 단계의 전문성이 희석됩니다. 역할 경계 없이 진행된 결과물은 일관성을 잃기 쉽습니다.

→ 기획·설계·실행·리뷰 각 단계를 전문 에이전트가 독립 담당합니다
🎲

확인 없이 코드를 만들어낸다

로컬 코드베이스나 공식 문서를 확인하지 않고 답변을 생성합니다. 그럴듯하지만 실제와 맞지 않는 코드가 나오는 이유입니다.

→ Explore와 Librarian이 증거를 먼저 수집하고 구현에 전달합니다
💨

중요한 결정이 채팅 로그에 묻힌다

핵심 맥락과 결정이 대화 기록으로만 남습니다. 다음 작업이 시작될 때 맥락을 다시 설명해야 합니다.

→ 모든 단계가 prd.md, design.md, execution-plan.md로 남습니다

Agent System

역할 기반 에이전트 분업

단일 AI가 모든 것을 처리하지 않습니다. 각 단계를 전담하는 에이전트가 역할을 맡아, 맥락 오염 없이 기획부터 리뷰까지 처리합니다.

Phase 1 · Planning 🧭

Mate

아이디어를 가져오면 PRD와 레퍼런스로 정리해 줍니다. 무엇을 만들지 명확히 결정되지 않았을 때 가장 먼저 부르세요.

📝 prd.md 🔗 references.md
Phase 2 · Design 🎨

Designer

PRD를 화면 설계 문서(design.md)로 변환해 줍니다. 구현 전에 레이아웃, UX, 시각 방향을 문서로 잠가 줍니다.

🖼 design.md
Phase 2 · Technical 🏗

Architector

PRD를 기술 명세(technical.md)로 변환해 줍니다. 아키텍처, 데이터 계약, 스택 결정을 구현 전에 정리해 줍니다.

⚙️ technical.md
Phase 4 · Review 🔍

Reviewer

보안, 코드 품질, 성능, 접근성을 독립적으로 점검합니다. 병합 전 최종 verdict를 역할 기반으로 내려 줍니다.

✅ review verdict
Phase 3 · Execution
⚡

Commander

실행 플랜을 세우고 진행 상황을 todo 기반으로 추적합니다. 설계가 끝나면 여기서 구현이 시작됩니다.

📋 execution-plan.md
🤖

Deep Execution Agent

Commander가 위임한 scope 안에서 실제 코드를 작성하고 검증 증거를 생성합니다. 구현 작업을 단계별로 처리합니다.

💻 implementation 🔬 verification
Support · Intelligence
🔭

Explore / Librarian

로컬 코드베이스와 외부 공식 문서를 탐색해 근거를 모읍니다. 추측 없이 evidence로 구현을 뒷받침합니다.

📊 evidence
🤝

Coordinator

PRD의 명확성과 범위 규율을 독립적으로 검토합니다. 기획 단계의 품질 지킴이 역할을 합니다.

📌 council verdict

Use Cases

어떤 작업에 쓸 수 있나요?

기능 기획부터 버그 수정, 코드 리뷰, 리팩토링까지. 실제 개발 흐름에서 바로 꺼낼 수 있는 시나리오입니다.

💡

새 기능 기획

"아이디어가 있는데 PRD가 없다"

Mate에게 아이디어를 말하면, 사용자 관점 정리부터 scope 정의까지 PRD 초안을 바로 만들어 줍니다.

🐛

버그 수정

"재현은 되는데 원인을 모른다"

문제를 설명하면 Explore가 코드베이스를 탐색하고, Deep Execution Agent가 원인을 찾아 수정합니다.

🔍

코드 리뷰

"PR을 올렸는데 리뷰가 아쉽다"

PR diff를 넣으면 Reviewer가 보안, 성능, 접근성, 코드 품질 관점에서 독립적으로 리뷰합니다.

🔧

리팩토링

"코드가 복잡해져서 손대기 어렵다"

Explore가 기존 코드를 분석하고, Commander 및 Architector와 함께 구조 개선 방향을 제안합니다.

Getting Started

4단계로 시작할 수 있습니다

Fork 또는 Clone 한 번으로 에이전트 시스템을 내 프로젝트에 추가할 수 있습니다.

  1. 01

    저장소를 복사합니다

    GitHub에서 Fork하거나 직접 Clone합니다.

    git clone https://github.com/your-username/copilot-agent-harness
  2. 02

    핵심 파일을 내 프로젝트에 추가합니다

    AGENTS.md와 .github/ 폴더 전체를 프로젝트 루트에 복사합니다. 에이전트 시스템의 핵심 설정 파일입니다.

  3. 03

    VS Code Copilot Chat에서 대화를 시작합니다

    Copilot Chat 패널을 열고 Agent 모드에서 Mate를 선택해 대화를 시작합니다.

  4. 04

    기획부터 리뷰까지 에이전트가 역할을 맡습니다

    Mate가 플래닝을 시작하고, Designer, Architector, Commander, Reviewer가 각 단계를 이어받아 처리합니다.
